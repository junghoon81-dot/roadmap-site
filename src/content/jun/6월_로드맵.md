# 김과외 6월 로드맵 v1.0

> 6월 3주차 이후 실행 범위. 5월 Phase 1에서 축적된 데이터와 인프라 위에서 에이전트 시대 대비 + 안심결제 확대를 진행합니다.

**메인 이슈 3가지:**

**1. 에이전트 시대 대비 — 실행 레이어 설계.** 5월 체크인·코칭·프로필 데이터를 구조화하여, AI 에이전트가 김과외를 "위임해도 안전한 플랫폼"으로 인식할 수 있는 검증된 실행 레이어를 만든다.

**2. 중재 고도화 + 선생님 대시보드.** Phase 1의 수동 CS 중재를 반자동화하고, 선생님이 직접 수업 상태를 볼 수 있는 대시보드를 구축한다.

**3. 안심결제 전국 도입 + 요금 체계 개선.** 수도권 중심 안심결제를 전국으로 확대하고, 요금 역전 구조를 해결한다.

---

## 1. 에이전트 대비 — 실행 레이어 설계

### 1-1. Outcome Graph 최소 모델

> 문서에 엔터티와 관계가 없으면 API, KPI, 학습 루프, 외부 공개 기준이 모두 임의적이 됩니다.

**기본 엔터티 (11종):** `household`, `student`, `teacher`, `match`, `lesson_start`, `weekly_checkin`, `coaching_report`, `dispute_case`, `mediation_action`, `rematch`, `review`

**모든 이벤트 필수 속성 (5종):** `timestamp`, `actor`, `source`, `confidence`, `visibility`

매칭→첫수업→유지→만족도→이탈사유→재매칭 결과까지 연결된 검증 데이터가 outcome graph입니다. 에이전트가 김과외를 선택하는 기준은 "데이터가 풍부한가"보다 **"위임했을 때 실패 비용이 가장 낮은가"**입니다.

### 1-2. 해자 재정의

3단계 이모지 체크인과 코칭 리포트는 경쟁 플랫폼도 복제 가능합니다. 복제 불가능한 것은 **"어떤 문제에 어떤 개입을 했더니 실제로 유지·회복·재매칭이 어떻게 됐는가"**라는 장기 폐루프 데이터입니다.

**해자 = `verified longitudinal intervention outcomes`.** 외부에 보여줄 때 `n`, `최근성`, `응답 커버리지`, `신뢰구간`이 같이 나와야 합니다.

가장 큰 에이전트 시대 위협은 "김과외가 데이터 공급원으로 전락하는 것"입니다. 에이전트가 여러 플랫폼의 공개 데이터를 조합해 자체 shortlist를 만들면, 플랫폼 프론트는 약해집니다. 방어 전략의 중심은 **리스팅이 아니라 실행 레이어**입니다.

김과외를 거치지 않으면 얻을 수 없는 가치: 분쟁 조정, 교체 보장, 진행 리포트, 결제/안전장치, 후기 진위 검증, 온플랫폼 활동 이력, 재매칭 성과. 에이전트 시대에는 검색되는 플랫폼보다 **"위임해도 안전한 플랫폼"**이 더 강합니다.

### 1-3. 에이전트용 API 사전 정의

read-only API의 실제 인터페이스가 없으면 Phase 1~2에서 무엇을 수집할지도 역으로 정해지지 않습니다.

**최소 3개 API:**
- `search_tutors` — 조건 입력 → 후보 리스트 반환
- `compare_tutors` — 후보 간 차이 + 근거 반환
- `get_tutor_profile` — 프로필 + 운영 신호 + 공개 가능한 성과 신호

**응답 필수 필드:** `fit_reasons`, `trust_metrics`, `execution_metrics`, `economics`, `confidence`, `freshness`

**에이전트가 김과외를 선택하는 4가지 기준:**

| 기준 | 김과외가 제공할 데이터 |
|------|----------------------|
| **적합도** | 구조화된 프로필, 추천 이유 카드, taxonomy 기반 매칭 정밀도 |
| **신뢰도** | 검증된 리뷰, 세그먼트별 만족도, outcome graph |
| **실행 가능성** | `last_active_at`, `median_response_time`, `accept_rate`, `availability`, 교체 보장 정책 |
| **경제성** | 가격 범위 명시, take rate 구조, 환불/정산 기준 |

### 1-4. Canonical API 우선, MCP는 포장 방식

내부적으로는 **프로토콜 독립적인 canonical API + 권한 모델**을 먼저 만들고, 그 위에 MCP wrapper를 얹습니다. 웹 에이전트, 앱 에이전트, 자체 봇 모두 같은 사실 모델을 씁니다.

### 1-5. 위임 API — draft + human confirm 구조

OAuth 토큰만으로 과도한 대리가 가능합니다. 쓰기 API는 처음부터 `draft + human confirm` 구조로 설계합니다.

**Scope:** `search:read`, `compare:read`, `lead:create`, `match_request:create_draft`, `feedback:submit_draft`, `rematch:request_draft`. 감사 로그 + 사용자 확인 화면 필수.

### 1-6. 품질 게이트 — agent-ready 공급만 노출

프로필 입력률 3~7% 상태에서 AI 요약과 SEO 노출을 넓히면 빈 프로필이 먼저 노출됩니다. **agent-ready 공급만 먼저 노출합니다:** 핵심 필드 완성 + 최근 활성 + 가용성 확인 + 최소 신뢰 신호가 있는 선생님만 검색/요약 대상에 포함.

### 1-7. 3단계 유통 전략

| 단계 | 구체적 작업 | 시기 |
|------|-------------|------|
| **1단계: 발견** | 공개 웹/SEO/LLM 친화 랜딩. 구조화된 프로필과 추천 이유가 검색엔진과 AI 요약에 노출 | Phase 1~2와 동시 |
| **2단계: 조회** | read-only search/compare API 또는 MCP. 에이전트가 조건을 보내면 후보+판단 근거를 응답 | Phase 3 초기 |
| **3단계: 위임** | 사용자 명시 승인 하의 쓰기 액션. 매칭 요청, 피드백 제출, 재매칭 등 | Phase 3 후기 |

**에이전트 타임라인 현실 인식:**
- **2026 하반기 = 발견/비교.** 에이전트가 김과외를 발견하고, 선생님 정보를 조회하여 비교하는 단계.
- **2027 = shortlist/lead 생성.** 에이전트가 조건에 맞는 후보를 뽑아 학부모에게 제시하는 단계.
- **그 이후 = 부분적 위임.** 에이전트가 학부모 대리로 매칭을 요청하고, 결과를 추적하는 단계.

올해 KPI는 에이전트 GMV가 아니라 **AI 인용 노출, AI 유입 리드, AI 기반 shortlist 전환율**입니다.

## 2. Phase 2 — 중재 고도화 + 선생님 대시보드 + 판단 보조 완성

> 5월 Phase 1에서 시작한 체크인 기반 관계 유지 시스템을 고도화합니다.

### 2-1. 중재 고도화

- Phase 1에서 수동(CS 개입)으로 운영하던 중재 프로세스를 **반자동화**. 소프트 미스매치 유형별 표준 코칭 리포트 템플릿 + 선생님 피드백 전달 자동화 + 재매칭 연결 원클릭 처리.
- 체크인 데이터 축적으로 **"이 유형의 불만족은 코칭 리포트로 해결된다 / 선생님 교체가 답이다"**를 판단할 수 있는 근거 확보.
- 중재 성공률 추적: 코칭 리포트 후 회복율, CS 개입 후 유지율, 재매칭 성사율.

### 2-2. 선생님 수업 대시보드

선생님이 직접 자기 수업 상태를 볼 수 있는 화면. 코칭 리포트가 "지적"이 아니라 "이미 아는 데이터의 일부"가 됩니다.

- 주간 만족도 추이 (최근 4주, 그래프)
- 강점 TOP 3 (학부모가 "좋아요" 시 선택한 항목 집계)
- 개선 포인트 (반복 아쉬움 카테고리가 있으면 표시)
- 재이용 의향

강점이 먼저 보이니까 기분이 나쁘지 않습니다. "개념 설명 잘함 85%" 옆에 "숙제 관리 개선 필요"가 있으면 공격이 아니라 균형 잡힌 정보. Phase 1에서 데이터가 충분히 쌓인 후에 만드는 것이 현실적입니다.

### 2-3. 판단 보조 시스템

- **공고 등록 즉시 AI 매칭 추천:** 자연어검색 API + 추천 이유 카드 v2 + 플레이빙 데이터 + 체크인 성과 데이터 조합. 매칭 시간 "며칠 → 즉시" 단축.
- **추천이유카드 v2-B:** 체크인 데이터가 충분히 쌓인 후, "비슷한 조건 학생 47명 기준 만족도 4.3/5.0" 형태의 성과 근거 추가. 세그먼트별 N이 기준 이상, 최근 6~12개월, confidence 표시 포함.

## 3. Phase 3 — 에이전트용 서비스 오픈

> Phase 1~2에서 쌓인 구조화된 데이터를 외부 에이전트가 접근할 수 있는 인터페이스로 개방합니다. 7~8월 성수기 이후 시작.

§1의 3단계 유통 전략(발견→조회→위임)을 순차 실행합니다.

**전제 조건:** Phase 1의 체크인 데이터 + taxonomy가 충분히 축적되어야 에이전트에게 줄 "판단 근거"가 있습니다. 데이터 없이 API만 열면 빈 껍데기. 동시에, API가 있어도 발견되지 않으면 존재하지 않는 것과 같습니다. **데이터 축적과 유통 채널 준비를 병행해야 합니다.**

## 4. Phase 4 — 관리형 과외 워크플로 실험 (장기)

Phase 1의 체크인은 "매칭 이후 관계를 끊지 않는 것"이고, Phase 2의 중재 고도화는 "문제가 생겼을 때 해결하는 것"입니다. Phase 4는 그 다음 단계 — **수업이 진행되는 동안 적극적으로 관리하는 것**입니다.

**구체적으로 무엇을 하는가:**
- **출결 체크:** 수업 예정일에 "오늘 수업 하셨나요?" 알림톡. 미응답 시 CS 확인.
- **숙제 로그:** 선생님이 숙제를 내면 학부모에게 "이번 주 숙제: OO" 알림. 이행 여부 기록.
- **학부모 리포트:** 월 1회 "이번 달 수업 N회 진행, 숙제 이행률 M%, 선생님 코멘트: ..."
- **목표 추적:** Phase 1 체크인에서 설정한 목표 대비 진행 상황 시각화.
- **이상 징후 시 재매칭 개입:** 2주 이상 수업 미진행, 학부모 레드 상태 등 감지 시 "다른 선생님 찾아볼까요?" 선제 제안.

**왜 Phase 4인가 — Phase 1~3이 선행되어야 하는 이유:**
- 체크인 데이터와 taxonomy가 쌓여야 "이상 징후"가 무엇인지 정의할 수 있습니다.
- 프로필이 충실해져야 "이 선생님의 관리 스타일"을 학부모에게 설명할 수 있습니다.
- 에이전트 API가 있으면 AI 비서가 학부모 대신 리포트를 읽고, 재매칭 판단을 보조할 수 있습니다.

**매출 모델:** "관리형 과외 프리미엄"으로 take rate 인상(기존 수수료 + 관리 수수료) 또는 월정액 관리비. 초기에는 소수 학부모 대상 유료 베타로 시작하여, 수업 유지율(LTV) 개선 효과를 검증한 후 확대합니다.

> **한 줄:** 매칭 플랫폼 → 체크인 기반 관계 유지 → 코칭 리포트 + 중재 + 판단 보조 → 에이전트 실행 레이어 → 관리형 과외 워크플로

## 5. 안심결제 전국 도입 + 요금 체계 개선

### 5-1. 안심결제 전국 확대
- 현재 수도권 중심 → 전국 주요 도시로 확대
- 선생님 온보딩 프로세스 간소화
- 결제 안정성 및 정산 주기 개선

### 5-2. 요금 역전 문제 해결
- 안심결제 이용요금이 일반결제보다 낮은 현상 분석
- 요금 체계 재설계 방안 수립
- 기존 이용자 전환 시나리오

## 6. Phase 전환 기준

달력이 아니라 **조건**으로 전환합니다.

| 전환 | 전제 조건 |
|------|-----------|
| Phase 1 → Phase 2 | 프로필 핵심 3필드 커버리지 목표 달성, 세그먼트별 표본 수 확보, 응답 커버리지 40%+, CS 처리 SLA 안정화 |

| Phase 3 → Phase 4 | 에이전트 API read-only 운영 안정, taxonomy 카테고리 안정화(2,500+ usable labels), 관리형 수요 검증 |
