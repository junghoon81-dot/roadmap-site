## 1. 에이전트 시대에 김과외는 어떻게 장악하는가

### 시장이 바뀐다

가까운 미래에 학부모는 직접 플랫폼을 비교하지 않습니다. 학부모의 AI 에이전트가 여러 플랫폼을 동시에 탐색하고, 가장 신뢰할 수 있는 근거를 가진 곳에서 매칭을 진행합니다.

이때 살아남는 플랫폼의 조건은 세 가지입니다.

**첫째, AI가 읽어도 의미 있는 구조화된 판단 근거를 가지고 있을 것.** "선생님 프로필"은 누구나 만들 수 있습니다. 하지만 "비슷한 조건의 학생 50명 중 43명이 이 선생님과 3개월 이상 수업을 유지했습니다"는 데이터가 있어야만 가능합니다. 에이전트는 이런 구조화된 판단 근거를 파싱합니다.

**둘째, 매칭 결과에 대한 피드백 루프가 돌고 있을 것.** 매칭이 왜 성공했고 왜 실패했는지를 자동으로 분류하고, 다음 매칭에 반영하는 시스템. 에이전트는 "이 플랫폼이 학습하고 있는가"를 판단합니다. 정적인 프로필 DB는 에이전트에게 매력이 없습니다.

**셋째, 에이전트가 직접 위임 설정을 전달할 수 있는 구조화된 인터페이스가 있을 것.** "중2 수학, 관리형, 주 2회, 강남, 예산 30만원" — 이런 위임 설정을 에이전트가 API로 전달하면 즉시 최적 매칭이 나오는 구조.

### 김과외가 이걸 할 수 있는 이유

김과외는 이미 핵심 자산을 가지고 있습니다.

| 자산 | 규모 | 에이전트 시대 가치 |
|------|------|-------------------|
| 누적 매칭 이력 | 492,106건 | 매칭 성공/실패 패턴 학습의 원천 데이터 |
| 누적 리뷰 | 125,527건 | 판단 근거의 시드 데이터 |
| 등록 선생님 | 650,205명 | 에이전트가 탐색할 공급 풀 |
| AI 맞춤 검색 | **4/13 배포 예정** | 자연어 → 매칭 스코어링 파이프라인 존재 |
| 알림톡 인프라 | 188개 템플릿 | 데이터 수집 채널 이미 구축 |
| SEO 랜딩 10만+ 페이지 | **4월 말 배포 예정** (Phase 0) | 에이전트가 크롤링할 구조화된 공개 데이터 |
| 매출향상 장치 8개 | **4월 배포 예정** (Phase 0) | 성사누락·휴면·FOMO 등 GMV 회수 인프라 |

**문제는 이 자산이 "구조화"되어 있지 않다는 것입니다.** 프로필 핵심 필드 입력률 3~7%, 거절 사유 로그 26건, 수업 성과 데이터 0건. 데이터는 쌓여있지만 판단 근거로 활용할 수 있는 형태가 아닙니다.

### 이번 패치의 본질

**이번 패치는 김과외의 기존 자산을 "에이전트가 읽을 수 있는 판단 시스템"으로 구조화하는 핵심 단계입니다.**

Phase 0(4월)에서 매출향상 장치와 SEO 랜딩을 먼저 깔고, Phase 1(5월 3주차~)에서 그 위에 판단 인프라를 올립니다.

구체적으로:
- 선생님 프로필을 채워서 매칭 재료를 풍부하게 만든다 → 에이전트가 파싱할 수 있는 구조화된 프로필
- 매칭 결과에 대한 피드백 데이터를 본격 수집한다 → 에이전트가 신뢰할 수 있는 판단 근거
- 매칭 실패/성공 이유를 분류하는 체계를 만든다 → 에이전트가 "이 플랫폼은 학습한다"고 판단하는 피드백 루프
- 학부모의 요구사항을 구조화한다 → 미래에 에이전트가 전달하는 "위임 설정"과 동일한 데이터 구조

**그리고 이 모든 작업은, 에이전트 시대가 오기 전인 지금 당장에도 효과가 있습니다.**

## 2. 에이전트 시대 준비가 왜 지금의 매출도 올리는가

학부모가 김과외에 돈을 내는 이유는 "선생님 목록"이 아닙니다. 학부모가 진짜 사는 건 **"우리 아이에게 이 선생님이 맞을 거라는 확신"**입니다.

에이전트 시대를 위해 쌓는 판단 근거 데이터는, 지금은 학부모에게 확신을 주는 재료로 쓰입니다.

| 에이전트 시대를 위해 하는 일 | 지금 당장의 효과 |
|---------------------------|----------------|
| 선생님 프로필 구조화 (③) | 매칭 품질 즉시 개선 → 성사율 ↑ |
| 설문 체계 전 구간 확장 (①) | 재매칭 유도 → 재이용률 ↑ / "관리하는 플랫폼" 신호 |
| 추천 이유 카드 고도화 (②) | "왜 이 선생님인지" 확신 → 문의 전환율 ↑ |
| 학부모 요구사항 구조화 (④) | 매칭 정밀도 ↑ → 미스매칭 ↓ |
| 매칭 실패/성공 분류 (⑤) | 다음 매칭이 더 정확해짐 → 이탈률 ↓ |

**매출 = 유입 × 성사율 × 건당 수수료 × 재이용률.** Phase 0의 매출향상 8개가 유입과 성사 누락을 잡고, Phase 1의 개편이 성사율과 재이용률을 구조적으로 끌어올립니다.

### 부수 효과: 성수기 극대화

김과외의 연중 매출 피크는 7~8월(여름방학)입니다 (25년 7월 9.72억/13,491건). 4월부터 매출향상 장치를 깔고 5월 3주차부터 판단 인프라를 올리면, 성수기에 유입이 몰릴 때 그 효과가 곱해집니다. 이것은 목적이 아니라, 본질대로 준비했을 때 자연스럽게 얻는 보너스입니다.

## 5. 실행 범위 — 핵심 기능 5개

### ① 설문요청 전 구간 확장

**에이전트 시대 의미:** 에이전트가 김과외를 신뢰하려면 "이 플랫폼은 매칭 결과를 추적하고 있다"는 증거가 필요합니다. 설문 데이터는 그 증거의 시작점입니다. 매칭 성과를 구조화된 형태로 수집하면, 미래에 에이전트가 "이 선생님의 비슷한 조건 학생 만족도"를 조회할 수 있는 API의 원천이 됩니다.

**지금 당장의 효과:** 설문 자체가 재이용 유도 채널이 됩니다. 만족도 높으면 "다른 과목도?" 유도, 낮으면 "더 잘 맞는 선생님 찾아드릴까요?" 제안. "설문한다는 것 자체가 우리 서비스가 관리하고 있다는 신호"이고, "기존 고객들을 다시 awake하는 효과"입니다.

**Phase 1 시작 시점 상태:** Phase 0에서 ⑥-H(성사 3개월 후 설문)가 배포되어 축적이 시작된 상태. 하지만 수업 초기·중기·종료 시점의 데이터는 없음.

**확장 범위 — 4시점 전 구간 설문:**

| 시점 | 질문 핵심 | 트리거 | 상태 |
|------|-----------|--------|------|
| 수업 2주 후 | 만족도 + 선생님 강점 | created_at + 14일 (또는 첫 수업일 + 14일) | **신규 개발** |
| 수업 1개월 후 | 만족도 + 지속의향 + 보완점 | created_at + 30일 | **신규 개발** |
| 수업 2개월 후 | 목표달성도 + NPS | created_at + 60일 | **신규 개발** |
| 수업 3개월 후 | 성적변화 + 종료이유 + 후기유도 | created_at + 90일 (단기 60일) | ⑥-H (4월 말 배포 예정) → Phase 1에서 통합 |
| 수업 종료 시 | 종료 사유 | status = 'completed' | **신규 개발** |

> 수업일정 알리미(Phase 0에서 배포 예정, 알림톡 방식)에서 "첫 수업일" 데이터가 축적되면, "2주 후" 트리거를 created_at 대신 실제 첫 수업일 기준으로 설정할 수 있습니다. 더 정확한 타이밍. (푸시가 아닌 알림톡: 푸시 설정을 꺼두거나 앱을 삭제한 분들도 수신 가능)

**모든 설문 완료 후:**
- 만족도 4~5점: "다른 과목도 찾고 계신가요?" → 재매칭 유도
- 만족도 3점: "어떤 부분이 아쉬우신가요?" → 추가 피드백 수집
- 만족도 1~2점: "더 잘 맞는 선생님을 찾아드릴까요?" → 재매칭 제안

**수집 항목:** 수업 만족도(1~5), 선생님 강점(복수선택: 숙제관리/개념설명/동기부여/시험대비/진도관리/커뮤니케이션), 수업 지속 의향, 미지속 사유(복수선택: 목표달성/비용/선생님불만족/시간/기타), 자유 의견(선택)

**발송 방식:** 카카오 알림톡 (Phase 0 매출향상 8개에서 검증될 알림톡 인프라 재활용) → 클릭 → 자동 로그인 → 간편 응답 화면 (탭 3~4번이면 완료)

**기존 후기 알림톡과의 관계:** 기존 CRM3/4/5 후기 요청 스케줄러는 `Program.cs`에서 전부 주석 처리되어 비활성 상태. ① 설문요청이 이를 완전히 대체합니다.

**개발 범위:**
- DB: lesson_survey 테이블 확장 (⑥-H 기존 테이블이 있으면 시점 필드 추가, 없으면 신규)
- Server: 설문 발송 cron job (매일 1회, 대상 매칭 스캔, 5시점 필터) + 설문 응답 API
- Web/App: 설문 응답 UI (웹뷰 or 네이티브) + 알림톡 딥링크
- 알림: kakao_template에 4개 시점별 신규 템플릿 등록

### ② 추천 이유 설명 카드 고도화

**에이전트 시대 의미:** 이 카드의 포맷이 곧 에이전트가 파싱하는 "판단 로그"의 공개 인터페이스입니다. "이 플랫폼의 추천 기준이 구조화되어 있다"는 신뢰 신호. 지금은 학부모가 보는 것이고, 나중에는 에이전트가 읽는 것입니다. **Phase 0에서 배포되는 SEO 랜딩 페이지에서도 이 카드가 노출되면 구글 크롤러 + 에이전트가 동시에 파싱합니다.**

**지금 당장의 효과:** "왜 이 선생님이 맞는지"에 대한 근거를 제시하여 문의 전환율 개선.

**2단계 고도화:**

**v2-A (이번 패치 범위):** ③에서 선생님 프로필이 채워지면 즉시 반영
- 프로필 기반 매칭 근거 강화: "이 선생님은 숙제관리 방식이 [상세]"
- 학부모 플레이빙(④)과 선생님 프로필(③)의 교차 매칭: "학부모 요청 [관리형] ↔ 선생님 스타일 [매주 숙제+피드백]"
- 기존 리뷰 125,527건에서 선생님별 강점 키워드 자동 추출
- **SEO 랜딩 페이지에 추천이유카드 v2 반영** → 비로그인 페이지에서도 판단 근거 노출

**v2-B (이번 패치 이후, ① 데이터 축적 후):** 설문 데이터 기반 성과 근거 추가
- "비슷한 조건 학생 47명 기준 만족도 4.3/5.0"
- 실제 수업 성과 사례

### ③ 선생님 프로필 입력률 끌어올리기

**에이전트 시대 의미:** 에이전트가 김과외의 선생님 풀을 탐색할 때, 프로필이 비어있으면 매칭 대상에서 제외됩니다. 프로필 입력률 7%는 에이전트 입장에서 "이 플랫폼은 데이터가 없다"는 뜻입니다. 구조화된 프로필은 에이전트가 읽을 수 있는 공급 카탈로그입니다.

**지금 당장의 효과 (배수 효과):**
- AI 자연어검색 정밀도 향상 (프로필 데이터가 검색 대상)
- SEO 랜딩 페이지 콘텐츠 풍부화 (Phase 0에서 프로필 비로그인 공개 예정이므로, 데이터가 채워지면 자동으로 SEO 페이지 품질 ↑)
- 추천이유카드 v2에 쓸 재료 확보
- 선생님 부스트(⑥-F) 팝업과 연계하여 노출 우선 인센티브

**핵심 실행 방법 — 선생님마다 안 채운 항목을 일 1회 덮는 화면으로 유도:**

김과외 선생님들은 패치 이후 빠르게 정보를 채워주시는 편입니다. **선생님 개인별로 아직 안 채운 프로필 항목을 파악하여 일 1회 덮는 화면(오버레이)으로 채우도록 유도**합니다.

- 선생님 앱/웹 접속 시 → 해당 선생님의 빈 프로필 항목 조회 → 빈 항목이 있으면 전체 화면 안내
- **매일 1개 항목씩 안내** (하루에 전부 보여주면 이탈). "오늘은 '숙제 관리 방식'을 채워볼까요?"
- isNew 뱃지 필드(입력률 최저)를 우선 안내하되, 기존 필드(차별점/어필 등)가 비어있으면 함께 안내
- 전부 채운 선생님에게는 덮는 화면 미노출
- 스킵 가능하되, 다음 접속 시 다른 빈 항목으로 재안내
- 모든 항목 완료 시 축하 메시지 + "프로필 완성도 100%" 뱃지

**보조 수단:**
1. 공고 지원 시점 유도: "이 정보를 입력하면 추천 확률이 올라갑니다" 메시지
2. 선생님 부스트(⑥-F) 연계: "프로필 완성도 70% → 90%로 올리면 노출 우선"
3. 프로필 완성도 시각화: 마이페이지에 진행률 바 + 빈 항목 명시
4. **SEO 연계 안내:** "프로필을 채우면 구글 검색에서도 선생님이 노출됩니다" (SEO 랜딩 배포 후 가능한 메시지)

### ④ 학부모 플레이빙 강화

**에이전트 시대 의미:** 지금 학부모가 입력하는 플레이빙 데이터는, 미래에 에이전트가 김과외에 전달하는 "위임 설정"과 구조가 같습니다. "중2 수학, 관리형, 주 2회, 강남" — 지금은 학부모가 직접 입력하지만, 나중에는 에이전트가 API로 전달합니다. 데이터 구조를 지금 잡아두면 전환이 자연스럽습니다.

**지금 당장의 효과:** 매칭 정밀도 향상. 미스매칭 감소.

**과제 1 — 공고개편에서 추가되는 필드 활용:** 4월 공고개편으로 수업목적(5종)·현재수준(5등급)·목표 3개 필드가 신규 추가됩니다. 이 데이터는 ②추천이유카드와 ⑤taxonomy에 직접 활용됩니다. 기존 필드(tutee_goal, specialty_point 등)의 입력률도 스텝 방식 전환으로 개선될 것으로 기대됩니다.

**과제 2 — Phase 1에서 추가할 필드:**
- **관리 강도 선호 (자율형 / 관리형 / 강한관리형):** 선생님의 homework_assignment_method와 매칭
- **이전 과외 경험 (처음 / 있음-만족 / 있음-불만족):** 불만족 사유를 함께 받으면 같은 실수를 반복하지 않는 매칭 가능. ⑤ taxonomy와 직결

**미성사 공고 재활용:** 기존 마감된 미성사 공고의 학부모에게 알림톡: "이 정보만 추가해주시면 더 잘 맞는 선생님 찾아드릴게요." 유입 비용 0원으로 추가 매칭 시도. (⑥-A 성사누락 잡기 인프라와 연계)

### ⑤ 매칭 실패/성공 이유 Taxonomy

**에이전트 시대 의미:** 이것이 피드백 루프의 핵심입니다. 에이전트가 "이 플랫폼은 매칭 결과를 학습하는가?"를 판단할 때, taxonomy가 있느냐 없느냐가 결정적입니다. 정적인 프로필 DB와 학습하는 매칭 시스템의 차이입니다.

**지금 당장의 효과:** 매칭 정확도 점진 개선. "강남 중2 수학에서 실패 원인 1위가 예산 불일치"라면 가성비형 선생님을 먼저 추천.

#### 갈래 1: 앞으로 쌓는 데이터 — 서비스 내 3개 시점 선택지

**시점 1 — 선생님 제안 거절 시:**
경력/학력 부족 | 예산 불일치 | 지역 불일치 | 시간 불일치 | 프로필 부족해서 판단 어려움 | 그냥 넘김

> DB 현황: lesson_offer_reject_log 현재 26건. 선택지가 깔리면 급격히 축적.

**시점 2 — 공고 마감 시:**
좋은 선생님 찾음 | 맞는 선생님 없음 | 다른 곳에서 찾음 | 과외 안 하기로 함

**시점 3 — 설문요청(①)에서 수업 종료 사유:**
목표 달성 | 비용 부담 | 선생님과 안 맞음 | 시간 안 맞게 됨 | 학원으로 변경

**→ 매칭 전(제안 거절) → 매칭 종료(공고 마감) → 수업 중/후(설문) 전 구간의 판단 데이터가 자동으로 쌓입니다.**

#### 갈래 2: 기존 데이터 영끌 — 매칭된 수업의 결과 데이터를 박박 긁기 ⭐ 4월 선행

> **핵심 관점: 매칭이 성사된 수업이 그 이후 어떤 결과로 이어졌는지를 추적하는 것.** 49만 건의 매칭 이력이 있지만, 매칭 이후 수업이 잘 됐는지 안 됐는지의 결과 데이터가 구조화되어 있지 않습니다. 이 결과 데이터를 영끌하여 "어떤 조건의 매칭이 좋은 결과로 이어지는가"를 알 수 있게 만듭니다.

> **이 작업은 5월 3주차 개편을 기다리지 않고 4월 13일부터 즉시 시작합니다.** 개발이 필요 없는 운영+데이터 작업이므로, 현욱님 3파트 패치와 병렬로 진행합니다.

**A. 매칭 이력 + 후기/평판 데이터 정제 (4월 즉시 착수):**
- 매칭 이력 492,106건에서 각 매칭이 어떤 결과로 이어졌는지 추적: 성사율, 유지 기간, 중도 이탈률, 재매칭 여부
- **후기/평판 시스템 데이터 정제:** 누적 리뷰 125,527건을 매칭 이력과 연결. 각 리뷰가 어떤 매칭에서 나온 것인지 매핑 → "이 조건의 매칭에서 만족도가 높았다/낮았다"를 역추적
- 낮은 평점 리뷰 키워드 분류 → 불만족 매칭의 공통 패턴 도출

**B. 대표님 데이터 조각 모음 (4/20~4/26 집중, 대표님 직접):**
- CS 상담 내역·슬랙 보고·후기/평판 데이터를 매칭 건과 연결하여 정제 **(대표님 직접)**
- 기존 매칭 내역 중 불만족으로 끝난 케이스들을 특정 → 해당 케이스와 관련된 CS 상담 내용을 찾아서 연결
- 슬랙 내부 채널(CS 보고, 운영 이슈 등)에서 매칭 불만족과 관련된 보고 패턴 추출
- **목적:** "이 매칭이 왜 불만족으로 끝났는지"의 구체적 맥락을 확보. 숫자(평점)만으로는 보이지 않는 실패 원인을 CS 상담 내용에서 파악.

**C. 운영팀 수동 수집 — CS/슬랙/후기로 데이터가 없는 매칭 건 영끌 (4/13~ 계속):**

매칭 건들 중 고객센터 상담 내역도, 슬랙 보고도, 후기/평판 데이터도 없는 건들이 대다수입니다. 이 "데이터 공백 매칭 건"들의 결과를 수동으로 수집합니다.

- **현욱님이 이번 주(4/7~) 수동 수집 프로세스를 설계합니다.** 어떤 매칭 건을 우선 접촉할지, 전화 시 어떤 질문을 할지, 수집된 데이터를 어떤 형식으로 정리할지.
- **4/13부터 운영팀 전담 2명이 매일 이 작업만 수행.** 학부모에게 직접 전화하여 "수업이 어떻게 됐는지" 확인. 특히 중도 이탈한 매칭의 학부모에게 "왜 그만두셨는지" 파악.
- **건수 목표를 잡지 않음. 할 수 있는 최대한 많이.**
- 이 작업은 Phase 0 기간 내내, 그리고 Phase 1 이후에도 계속됩니다.

**D. ⑥-A/B에서 쌓이는 데이터 활용 (4월 패치 배포 후 자동 축적):**
- 성사누락 알림톡(A)에서 "수업 안 시작한 사유" 데이터
- 시범→정규 미전환(B)에서 "전환 안 한 사유" 데이터
- 매출향상 8개 배포 후 자동으로 축적 → 5월 3주차에 시드로 활용

> **요점:** 자동 수집 시스템이 완성되기를 기다리지 않고, **4월 13일부터** 가능한 모든 방법으로 "매칭 → 수업 → 결과"의 데이터를 끌어모은다. 개발 대기 시간 0.

**쌓인 데이터로 할 수 있는 것:**

### Phase 2: 판단 보조 시스템 완성 (6월 3주차~)

- **공고 등록 즉시 AI 매칭 추천:** 자연어검색 API + 추천 이유 카드 v2 + 플레이빙 데이터 + 설문 성과 데이터 조합. 매칭 시간 "며칠 → 즉시" 단축.
- **② 추천이유카드 v2-B:** ① 설문 데이터가 충분히 쌓인 후, "비슷한 조건 학생 47명 기준 만족도 4.3/5.0" 형태의 성과 근거 추가.

### Phase 3: 에이전트용 서비스 오픈 (7~8월 성수기 이후)

Phase 1~2에서 쌓인 구조화된 데이터를 외부 에이전트가 직접 접근할 수 있는 인터페이스로 개방합니다.

**유저 입장에서 어떻게 되는가 — 예시:**

학부모가 AI 비서(오픈클로, 클로드 등)에게 "우리 아이 중2인데 수학 과외 알아봐줘. 관리형으로, 주 2회, 강남, 30만원 정도"라고 말합니다.

AI 비서가 김과외 API를 호출합니다. 김과외가 "이 조건에 맞는 선생님 3명"과 함께 **판단 근거**를 돌려줍니다: "A 선생님 — 비슷한 조건(중2 수학, 관리형) 학생 47명 기준 평균 유지기간 8개월, 만족도 4.3/5.0. 숙제관리 방식: 매주 문제집 지정+풀이 사진 확인." AI 비서가 이걸 학부모에게 설명합니다. 학부모가 "A 선생님으로 해줘"라고 하면, AI 비서가 김과외에서 매칭을 진행합니다.

**이게 가능하려면 필요한 것:**
- 선생님 프로필이 채워져 있어야 함 → Phase 1 ③
- 매칭 결과 데이터가 쌓여 있어야 "47명 기준 만족도 4.3"을 말할 수 있음 → Phase 0 영끌 + Phase 1 ①⑤
- 구조화된 API가 있어야 AI 비서가 호출할 수 있음 → Phase 3에서 개발

**에이전트 API:**
- 매칭 위임 API: 조건 입력 → 후보 + 판단 근거 응답
- 선생님 검증 데이터 API: 매칭 성과(유지기간, 만족도, 비슷한 조건 학생 성과) 조회
- 매칭 피드백 API: 에이전트가 매칭 결과를 보고 → 다음 매칭 정밀도 향상

**전제 조건:** Phase 1의 설문 데이터 + taxonomy가 충분히 축적되어야 에이전트에게 줄 "판단 근거"가 있음. 데이터 없이 API만 열면 빈 껍데기.

### Phase 4: 매칭 이후까지 관여하는 플랫폼 (장기)

Phase 1~3은 "매칭 전"과 "매칭 시점"의 데이터를 다룹니다. Phase 4는 **"매칭 이후 수업이 진행되는 동안"**까지 김과외가 관여하는 구조입니다.

- Phase 1에서 설문(①)으로 수업 결과를 수집하기 시작하면, "수업 도중에 문제가 생겼을 때 개입할 수 있으면 좋겠다"는 니즈가 자연스럽게 나옵니다.
- 수업 진행 체크, 숙제 이행률, 성과 리포트 같은 기능이 그 연장선입니다.
- 매출 구조도 바뀝니다: 매칭 1회 수수료 → 수업이 지속되는 동안 발생하는 가치(LTV).
- 과목 확장/업셀 ("수학 만족하셨으니 영어도?")도 수업 결과 데이터가 있어야 가능합니다.

**지금 하는 것과의 연결:** Phase 1의 설문 데이터와 taxonomy가 "매칭 이후 수업이 어떻게 됐는지"를 추적하는 첫 단추입니다. 이 데이터가 쌓여야 "수업 중 개입"이 의미 있고, 수업 중 개입이 되어야 LTV 모델이 가능합니다.

> **한 줄:** 검색 플랫폼 → 판단 보조 시스템 → 에이전트 오픈 플랫폼 → 매칭 이후까지 관여하는 플랫폼
